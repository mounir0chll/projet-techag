\chapter*{Introduction}
\phantomsection
\addcontentsline{toc}{chapter}{Introduction}
\markboth{Introduction}{Introduction}

Ce rapport présente le mini-projet de Technologies des Agents. Le travail demandé est découpé en quatre parties : une enchère JADE, un agent mobile avec décision multicritère, un test du planificateur POP d'AIMA et une simulation de planification distribuée Bill/Tom.
Les outils utilisés sont JADE pour les agents Java, AIMA-Python pour la partie POP, et Python pour la simulation Bill/Tom.

\begin{infobox}[Principe du rapport]
Le rapport ne redéfinit pas les notions vues en cours. Il montre l'implémentation réalisée, les paramètres utilisés et les sorties console qui prouvent l'exécution.
\end{infobox}

\chapter{Partie 1 -- Enchère JADE}

\badge{JADE}\quad\badge{ACL}\quad\badge{CFP}\quad\badge{PROPOSE}\quad\badge{Prix de réserve}

\section{Objectif}

Nous avons implémenté une enchère avec un \texttt{SellerAgent} et deux \texttt{BuyerAgent}. Le vendeur envoie un \texttt{CFP} à chaque round. Les acheteurs répondent par \texttt{PROPOSE} si le prix proposé reste sous leur maximum, sinon par \texttt{REFUSE}. Le vendeur garde la meilleure offre et termine par acceptation ou rejet.

\begin{querybox}[Point vérifié]
Le prix de réserve reste connu uniquement du vendeur. La vente n'est confirmée que si la meilleure offre atteint cette réserve.
\end{querybox}

\section{Structure des classes}

\begin{center}
\begin{tblr}{

    hline{2}={0.5pt,accent}
}
Classe & Rôle \\
\texttt{SellerAgent} & Pilote les rounds, filtre les réponses avec \texttt{conversationId} et \texttt{inReplyTo}, vérifie la réserve. \\
\texttt{BuyerAgent} & Répond aux \texttt{CFP}, propose \texttt{currentPrice + increment}, refuse si le prix dépasse \texttt{maxPrice}. \\
\texttt{AuctionLauncher} & Lance les scénarios \texttt{sold} et \texttt{unsold}. \\
\end{tblr}
\end{center}

\badge{Agent mobile}\quad\badge{doMove}\quad\badge{beforeMove}\quad\badge{afterMove}\quad\badge{Score pondéré}

\section{Score multicritère}

Nous avons implémenté l'exercice 5 du DW4 avec trois vendeurs. L'acheteur collecte le prix, la qualité et le coût de livraison, puis applique un score pondéré :

\[
score = 0.35 \times priceScore_{\min}
      + 0.50 \times qualityScore_{\max}
      + 0.15 \times deliveryScore_{\min}
\]

\section{Limite inter-plateformes}

Nous n'avons pas exécuté une vraie migration inter-plateformes complète. Pour la tester réellement, il faut deux plateformes JADE séparées, des adresses MTP accessibles, les mêmes classes déployées sur les deux JVM et une configuration de sécurité compatible.
Le code exécuté valide donc le cas inter-conteneurs.
